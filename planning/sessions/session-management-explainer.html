Session management explainer

How coding agents remember work

A single-page explainer comparing Pi, Claude Code, and OpenCode: how they save, load, replay, branch, switch models, run shell commands, and represent user/assistant/tool messages.

The big idea

A coding-agent “session” is more than chat history. It is an execution log: prompts, assistant output, tool calls, shell results, file changes, token/cost metadata, model choices, compaction summaries, and branch/rollback state.

01. Snapshot comparison

The three agents converge on continuous local persistence and resumable conversations, but they choose different storage and replay models.

Area Pi Claude Code OpenCode
Storage JSONL file per session, organized by working directory. JSONL transcript per project/session under ~/.claude/projects. SQLite-backed project/session/message storage under app data.
Resume pi -c, pi -r, --session, /resume. claude --continue, --resume, /resume, --from-pr. opencode -c, --session, /sessions, opencode session list.
Replay/branching Best native tree: id/parentId, /tree, /fork, /clone. Session branches via /branch or --fork-session; checkpoint rewind. Fork/child sessions and compaction sessions; undo/redo for message/file state.

02. Pi: sessions as editable JSONL trees

Storage and loading

Pi auto-saves sessions to ~/.pi/agent/sessions/, grouped by working directory. Each session is a JSONL file whose entries form a tree with id and parentId.

pi -c                  # continue most recent
pi -r                  # browse sessions
pi --session <path|id> # load specific session
pi --fork <path|id>    # fork to new file
pi --no-session        # ephemeral mode

Replay model

Pi is the clearest replay design. /tree can jump to any historical entry. Continuing from a previous user turn creates a new branch in the same file; /fork and /clone create separate files.

Key insight: Context is rebuilt by walking from the active leaf to the root, then applying compaction, branch-summary, custom-message, model, and thinking-level entries.

Message roles

user, assistant, toolResult, bashExecution, custom, branchSummary, compactionSummary.

Model switching

/model, /scoped-models, Ctrl+P cycling, --provider, --model, --thinking. Model changes are first-class session entries.

Shell commands

!command runs and sends output to context. !!command runs but marks output as excluded from model context.

03. Claude Code: project transcripts and branch sessions

Storage and loading

Claude Code continuously saves local JSONL transcripts at ~/.claude/projects/<project>/<session-id>.jsonl. Each line is a JSON object for a message, tool use, or metadata entry.

claude --continue
claude --resume
claude --resume <name-or-id>
claude --from-pr <number>
/resume

Session picker UX

The picker supports search, preview, rename, branch filtering, widening to worktrees or all projects, and grouped display of forked sessions. Named sessions can be set at startup or during a session.

  • Ctrl+W: all worktrees
  • Ctrl+A: all projects
  • Ctrl+B: current branch filter
  • Space: preview

Branch and rewind

/branch creates a new branch session. /rewind can roll conversation/code back to checkpoints or summarize from a selected message.

Model switching

/model, --model, ANTHROPIC_MODEL, and settings. Resumed sessions keep the model saved in the transcript unless retired.

Bash behavior

The Bash tool is permissioned. Commands have time/output limits; large output is saved to session files with a preview. Long-running commands can run in background.

Parser caution: Claude Code documents transcript JSONL location and broad content, but not a stable full public schema. Transcript consumers should be defensive and version-tolerant.

04. OpenCode: SQLite sessions with normalized messages and parts

Storage and loading

OpenCode persists session data, messages, tokens, costs, file changes, and metadata in app/project storage. Current docs place app data under ~/.local/share/opencode/; source shows SQLite with migrations and WAL mode.

opencode -c
opencode --session <id>
opencode --fork
opencode session list --format json
opencode export [sessionID]
opencode import <file-or-share-url>

Compaction lifecycle

OpenCode auto-compacts near context-window limits. It summarizes decisions, technical context, current state, and background, then creates a child session with the summary as initial context.

MonitorTrack tokens.
DetectApproach context limit.
SummarizeCreate durable summary.
SpawnContinue in child session.

Schema shape

Session table stores parent, title, model, cost, tokens, permissions, revert data. Messages and parts are separate rows with JSON data.

Message parts

Roles are user/assistant. Parts include text, reasoning, file, source URL, step-start, and tool invocation states.

Providers

OpenCode uses AI SDK and Models.dev for 75+ providers. /connect, /models, config model, and --model provider/model control selection.

05. Cross-agent patterns

What every serious coding-agent session needs

  • Durable user and assistant turns.
  • Structured tool calls and tool results.
  • Shell-command output, including truncation/full-output paths.
  • Provider/model/variant and token/cost accounting.
  • File-change metadata for undo, replay, or audit.
  • Compaction summaries and boundaries.
  • Branch/fork/parent metadata.

Storage tradeoffs

JSONL is transparent, append-friendly, human-inspectable, and easy to share. It is ideal for event sourcing and replay.

SQLite is queryable, transactional, scalable for lists/stats/multi-UI, and better for normalized message parts.

A robust design can use both: append-only event log as source of truth, SQLite as index/cache.

06. Recommended design for an Agento-like agent

1. Event log first

Store an append-only, versioned session event log with human-readable exports.

2. Explicit tree

Use id/parentId, labels, active leaf, fork metadata, and branch summaries.

3. Model timeline

Persist model/provider/thinking changes as events and stamp assistant outputs with model metadata.

4. Structured tools

Record call ID, tool name, args, output, errors, duration, truncation, and context inclusion.

5. Context vs display

Distinguish what the user sees from what the model receives, especially for hidden shell commands.

6. Import/export

Support machine-readable JSON plus Markdown/HTML, with sanitization for sharing.

Best combined approach: Pi's explicit replay tree + Claude Code's polished picker/retention controls + OpenCode's queryable SQLite metadata would make a strong session system.
