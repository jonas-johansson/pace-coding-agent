Parallel Tool Call Support

A plan for allowing Agento to run multiple independent tool calls at the same time, while keeping streamed UI updates, provider compatibility, cancellation, and tool-result ordering correct.

Anthropic OpenAI Responses API OpenAI-compatible Chat Completions TUI streaming Safe cancellation

What is the feature?

Today, when the model asks to use several tools in one turn, Agento handles them one by one. Parallel tool call support lets Agento execute independent tool calls concurrently. For example, reading three files or fetching several URLs can happen at the same time instead of waiting for each call to finish before starting the next one.

Before

Sequential: tool A finishes, then tool B starts, then tool C starts.

read file A ──────▶ done
                 read file B ──────▶ done
                                  web_fetch C ──────▶ done

After

Parallel: safe tools start together and results are returned together.

read file A ──────▶ done
read file B ───▶ done
web_fetch C ─────────▶ done

return all tool results in original order

Why this needs more than Promise.all

Running tools in parallel is the easy part. The hard parts are preserving stream identity, grouping results correctly for providers, handling interleaved tool argument deltas, and avoiding unsafe concurrent side effects.

1

Streaming identity

Tool argument deltas must say which tool call they belong to. Without an ID, interleaved deltas can update the wrong TUI block.

2

Grouped results

If an assistant message contains multiple tool uses, the next user message should contain all matching tool results together.

3

Side-effect safety

Reads and fetches are safe to parallelize. Writes, edits, and shell commands can conflict and may need sequential execution.

Core data model change

Tool streaming events should carry the tool-call ID. This makes the provider abstraction safe for models that emit multiple function calls at once. 

export type StreamToolInputDelta = {
  type: "tool_input_delta";
  id: string;
  partialJson: string;
};

export type StreamBlockStop = {
  type: "block_stop";
  id?: string;
};

Text blocks can still use a plain block_stop. Tool blocks should include id so the app can finish the correct streaming state.

Provider changes

Anthropic

Anthropic already supports multiple tool_use blocks. Track the current tool ID while streaming and attach that ID to input deltas and tool block stops.

mostly app-level work add IDs to deltas

OpenAI Responses

Enable parallel_tool_calls: true. The stream adapter already tracks calls by item ID, call ID, and output index; use that state to emit per-call deltas.

remove parallel disable route by call_id

OpenCode Zen

Chat Completions streams tool calls by index. Use stable IDs for each index, falling back to generated IDs if the provider does not send one early enough.

route by index stable fallback IDs

App-level execution flow

1. Stream the assistant response.

As tool calls appear, create TUI tool blocks and accumulate partial JSON per tool ID.

2. Collect all tool-use blocks.

After the stream completes, gather every tool_use block from the assistant message.

3. Execute the batch.

Run safe tools with Promise.all. Preserve result order by mapping over the original tool-use list.

4. Return one grouped user message.

Push a single user message containing all tool_result blocks, then continue the model loop.

Streaming UI state

The current app tracks only one active tool call. Parallel support should replace that with a map keyed by tool-use ID. 

type StreamingToolState = {
  name: string;
  inputJson: string;
};

const streamingTools = new Map<string, StreamingToolState>();

// On tool_use_start:
streamingTools.set(event.id, { name: event.name, inputJson: "" });

// On tool_input_delta:
const state = streamingTools.get(event.id);
if (state) {
  state.inputJson += event.partialJson;
  updateToolBlockTitle(event.id, state.name, state.inputJson);
}

Grouped tool results

Even though tools can finish in any order, results should be sent back to the model in the same order the assistant requested them. Promise.all preserves array order, so the result array remains predictable. 

const toolUseBlocks = response.content.filter(isToolUseBlock);

const toolResults = await Promise.all(
  toolUseBlocks.map((block) => executeToolUse(block, signal))
);

messages.push({
  role: "user",
  content: toolResults,
});

Schema validation failures, thrown errors, and tool-level failures should all become individual tool_result blocks with is_error: true. One bad call should not prevent the model from receiving results for the rest of the batch.

Concurrency safety policy

Some tools are naturally safe to run in parallel, while others can mutate files or run commands. Add a small concurrency hint to each tool descriptor. 

type ToolConcurrency = "safe" | "exclusive";

type ToolDescriptor = {
  name: string;
  description: string;
  inputSchema: ZodObjectSchema;
  execute: (...args) => Promise<ToolOutput>;
  concurrency?: ToolConcurrency;
};

Safe

  • read
  • web_fetch
  • web_search

Exclusive

  • write
  • edit
  • bash

Recommended first policy: if a batch contains any exclusive tool, execute the whole batch sequentially. Otherwise, run the batch in parallel.

Cancellation behavior

All parallel tool executions should share the same AbortSignal. When the user cancels, running tools receive the abort, pending TUI blocks are marked cancelled, and the conversation is repaired with error tool results for any unfinished calls. 

messages.push({
  role: "user",
  content: pendingToolUseIds.map((id) => ({
    type: "tool_result",
    tool_use_id: id,
    is_error: true,
    content: [{ type: "text", text: "Cancelled by user" }],
  })),
});

Implementation checklist

Update stream event types

Add tool-call IDs to input deltas and tool block stops in provider.ts.

Update provider adapters

Emit ID-aware tool events from Anthropic, OpenAI Responses, and OpenCode Zen.

Enable OpenAI parallel calls

Set parallel_tool_calls: true once deltas can be routed by ID.

Refactor app streaming state

Replace single active tool state with a Map<toolUseId, state>.

Execute tool batches

Use Promise.all for safe batches and sequential fallback for exclusive tools.

Group results

Push one user message containing all tool results in original assistant order.

Validate manually

Try multiple reads, multiple fetches, mixed safe/exclusive tools, cancellation, and OpenAI parallel streams.
