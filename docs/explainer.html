How Agento Works

A walkthrough of the architecture, data flow, and key design decisions of this barebones terminal-based AI coding agent.

🏗️ High-Level Architecture

Agento is a TypeScript Node.js application structured into three focused modules that collaborate to produce a fully interactive terminal agent:

Orchestrator
app.ts
~635 lines
⟷
Terminal UI
tui.ts
~2 453 lines
⟷
Tool Registry
tool.ts
~636 lines
  • app.ts — the brain. Manages the conversation loop, model selection, cost tracking, and wires together the TUI and tool execution.
  • tui.ts — the face. A from-scratch terminal renderer with ANSI escape codes, mouse support, scrolling, block-level caching, and line-level diffing.
  • tool.ts — the hands. Defines and registers every capability the AI can invoke: reading files, writing files, editing files, running bash commands, fetching web pages, and searching the web.

🚀 Startup Sequence

1
Terminal UI initialised

A Tui instance is created, wired to three callbacks: onSubmit (user sends a message), onTab (cycle models), and onEscape (cancel running prompt). Raw-mode stdin is enabled, the cursor is hidden, mouse reporting is switched on, and the ASCII logo is drawn.

2
Context gauge zeroed

updateContextInfo() populates the status bar with the initial 0 / 200k (0%) token counter for the default model.

3
User types and presses Enter

The TUI fires handleUserInput(message). If the message starts with / it is a slash command; if it starts with ! it is a direct shell command; otherwise it is sent straight to the AI.

🔄 The Agent Loop

When a user message reaches prompt() in app.ts, the following loop runs until the model signals it is finished:

1
Build system prompt

The base instruction ("You are Agento…") is assembled. If an AGENTS.md file exists in the current directory it is appended, giving per-project customisation. A cache_control: ephemeral breakpoint is placed on the system prompt so Anthropic's API can cache it cheaply across turns.

2
Stream the model's response

The SDK's ant.messages.stream() is called with the full conversation history, the registered tools (converted to Anthropic's JSON-schema format), and max_tokens: 16 000. As events arrive they are classified:

  • content_block_start — creates a new TUI block (text or tool)
  • text_delta — appends streamed text to the assistant block live
  • input_json_delta — updates the tool block's title as JSON input builds up (using partial-json to parse incomplete JSON)
  • content_block_stop — finalises the block
3
Execute tool calls

After the stream ends, any tool_use blocks in the response are executed sequentially. Each tool's input is validated through a Zod schema. If validation passes, the tool's execute() function runs with an AbortSignal so it can be cancelled mid-flight. Results (or errors) are pushed back into the message history as tool_result blocks.

4
Continue or stop

If stop_reason === "tool_use" the loop iterates — the model receives all the tool results and generates another response. Any other stop reason (typically "end_turn") exits the loop and hands control back to the user.

5
Token accounting & cost tracking

After each API call, response.usage is inspected to update the context gauge and accumulate the session cost using per-model pricing constants (input, cache-write, cache-read, and output tokens are all priced separately).

⛔ Cancellation (Esc)

Pressing Esc while the agent is running calls AbortController.abort(). All async paths — the streaming HTTP request, tool execution, and bash child processes — honour the same signal. The cleanup logic is careful to keep the message history valid:

  • If the model was still streaming (no assistant message committed yet), the pending user message is popped off the history.
  • If the model had committed an assistant message but tools weren't run yet, synthetic tool_result entries are injected for each outstanding tool_use so the API never sees an orphaned tool call.
  • The bash tool kills the entire process group (process.kill(-pid, SIGTERM)) so backgrounded children don't linger.

🔧 Available Tools

Tools are registered with the Tool() helper in tool.ts. Each tool carries a Zod schema (used both for runtime validation and for generating the JSON Schema sent to Anthropic), an execute() function, and an optional titleFormatter that produces a human-readable one-liner for the TUI.

read

Reads a file, emitting numbered lines. Supports offset and limit for pagination. Long lines are truncated at 2 000 chars; total output is capped at 50 KB.

showContent: false

write

Overwrites a file with the supplied content. Simple — calls fs/promises writeFile directly.

showContent: false

edit

Performs an exact-text find-and-replace inside a file using String.replaceAll(). The old text must match exactly (case-sensitive, whitespace-sensitive).

showContent: false

bash

Runs any shell command via spawn("/bin/sh", ["-c", command]) with a 10-second timeout. Uses detached: true so the full process group can be killed on abort or timeout.

showContent: true

web_fetch

Fetches a URL and returns it as markdown, text, or html. HTML is converted to Markdown via Turndown. Cloudflare challenge pages trigger a plain-UA retry.

showContent: true

web_search

Searches the web via the Exa MCP API (Server-Sent Events protocol). Returns a configurable number of results (default 5).

showContent: true

Tools tagged showContent: false hide their result body in the TUI (only the one-line title and a ✓ glyph are shown). Errors are always shown regardless of this flag.

⌨️ Input Handling

There are three special input prefixes that bypass the AI entirely:

Prefix Behaviour
/new Clears the message history and resets cost/token counters — starts a fresh conversation.
/model [id|alias] Without argument: lists available models. With argument: switches the active model immediately.
!<command> Runs a shell command directly using the bash tool, bypassing the AI entirely. Useful for quick one-off commands.

Keyboard shortcuts

💾 Prompt Caching

Agento uses Anthropic's ephemeral prompt caching to reduce cost on long conversations:

  • The system prompt always carries a cache_control: { type: "ephemeral" } breakpoint, so it is written to cache on the first call and read cheaply on every subsequent turn.
  • A top-level cache_control on the API request body causes Anthropic to automatically place a breakpoint on the last cacheable block in the growing conversation, keeping the ever-expanding history prefix in cache.
  • Cache statistics (cache read tokens and cache creation tokens) are surfaced in the status bar so you can see how effective the cache is in real time. A cache hit rate is shown as a percentage.

🖥️ The Terminal UI

The TUI is built from scratch using raw ANSI escape sequences — no curses library. The screen is divided into three zones, stacked vertically:

Let me check the project structure for you.
✓ bash: /bin/ls -1
The project has these main files: app.ts, tui.ts, tool.ts…
Can you read the README?
⠹ read: README.md
› Type a message…
⠋ Streaming response /home/user/project  |  $0.0023  |  4.2k/200k (2%)  |  claude-haiku-4-5
  • Message area — scrollable list of blocks, fills all rows not taken by the input and status.
  • Input box — dark background, supports multi-line editing, word-wrap, paste detection.
  • Status bar — spinner + status text on the left; cwd, cost, context gauge, model on the right.

Block types

User block

Blue boxed card with padding. Shows the raw message text with light Markdown parsing (bold, italic, inline code, headings).

Assistant block

Plain text on the dark canvas (no box). Rendered inline, with Markdown parsing. Streams live as tokens arrive.

Inline tool block

A single line on the canvas showing the tool name + arguments + a state glyph (spinner / ✓ / ✗). Used when the tool produces no visible output.

Panel tool block

A gray-background panel with content (e.g. bash output). Can be collapsed to 15 lines — click to expand. Shows a "Click to expand" hint when truncated.

Error block

Red panel for API errors, unknown commands, input schema failures, or uncaught exceptions. Always shows full content even if showContent is false.

⚡ Rendering Performance

The TUI is re-rendered on every state change (input keypress, incoming token, spinner tick). Several layered optimisations keep this fast even during rapid streaming:

P0 — Block-level cache

Each block's rendered lines are cached by (content, title, state, collapsed, columns). Only blocks that actually changed are re-rendered. Blocks in state "running" bypass the cache because the spinner changes every frame.

P1 — Line-level diff

The rendered frame is compared line-by-line to the previous frame. Only changed rows are written to stdout using absolute cursor positioning (\x1b[row;1H\x1b[2K…). Unchanged rows are skipped entirely.

P2 — Lazy screen lines

The per-row plaintext array (used for mouse/copy) is expensive to compute because it requires stripping ANSI codes from every row. It is only recomputed on the first mouse event after a frame change.

Batched renders

Multiple synchronous state mutations that happen in the same tick (e.g. updating multiple blocks) coalesce into a single render via setImmediate(), avoiding redundant frame computation.
